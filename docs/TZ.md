# ТЗ для Code LLM: Консольное приложение приоритизации источников
## 1. Контекст проекта
- **Тип проекта:** консольное CLI-приложение на Python 3.10+
- **Назначение:** пользовательская перекрёстная приоритезация источников информации, описанных в markdown-файлах с frontmatter
- **Стек:** Python 3.10+, стандартная библиотека (argparse, pathlib, re), без внешних зависимостей
- **Ключевые пути:** приложение работает с директорией, указанной в аргументе --path, где лежат .md-файлы источников

## 2. Текущая реализация и проблема
- Реализации нет — задача «с нуля»
- Проблема: ручная приоритезация большого количества источников неэффективна; требуется автоматизация с элементами пользовательского выбора
- Обратная совместимость: не требуется (новый инструмент)

## 3. Цель задачи
Создать консольное приложение на Python, которое:
1. Считывает markdown-файлы источников из указанной директории
2. Фильтрует непрочитанные источники (readed: false)
3. Позволяет пользователю выбрать тип источника для приоритезации
4. Реализует двухэтапный алгоритм приоритезации (групповой отбор + парные сравнения)
5. Обновляет файлы источников, добавляя маркеры приоритета в строку с названием

**Тип задачи:** новая функциональность, CLI-утилита

## 4. Требования к функционалу
### 4.1. Аргументы командной строки
| Аргумент | Обязательный | Описание |
|:---|:---|:---|
| `--path` | Да | Путь к директории с markdown-файлами источников |
| `--count` | Нет | При указании выводит в консоль список типов с количеством и завершить работу |

### 4.2. Парсинг файлов
- Читать все .md-файлы из указанной директории (рекурсивно или нет — по умолчанию не рекурсивно)
- Извлекать frontmatter-блок между --- в начале файла (YAML-like формат)
- Использовать поля:
    - `type`: тип источника (статья, книга, видео и т.д.)
    - `readed`: булево значение (`true`/`false`)
- Извлекать первую строку после frontmatter, начинающуюся с - [ ] или - [/] — это название источника

### 4.3. Фильтрация
- Обрабатывать только файлы с `readed: false`
- Файлы с `readed: true` игнорировать

### 4.4. Подсчёт и вывод статистики
- Подсчитать количество источников по каждому значению поля type
- Если указан флаг `--count`, вывести в консоль:
```
Непрочитанных источников:
Тип1: X
Тип2: Y
...
```

### 4.5. Выбор типа источника
- Вывести меню выбора типа:
```
Выберите тип источника для приоритезации:
1. книга (количество)
2. статья (количество)
...
0. Выход
```
- При выборе 0 — завершить работу без изменений

### 4.6. Подготовка к приоритезации
Для каждого источника выбранного типа в строке названия (`- [ ] ...` или `- [/] ...`):
- Удалить любые из значков приоритета в конце строки: 🔺, ⏫, 🔼, 🔽, ⏬
- Если строка начинается с `- [/]`, заменить на `- [ ]`

### 4.7. Алгоритм приоритезации (два этапа)
#### Этап 1: Групповой отбор (если источников > 50)
- Вычислить `alpha = ceil(count / 25)` — размер группы
- Разбить источники на группы по alpha элементов
- Для каждой группы:
    - Вывести меню выбора:
    ```
    Выберите приоритетный источник:
    1. название1
    2. название2
    ...
    0. Выход
    ```
    - Выбранный источник переходит на Этап 2
    - Остальные источники группы сразу получают статус lowest

#### Этап 2: Перекрёстная приоритезация (парные сравнения)
- Для всех источников, прошедших Этап 1:
    - Последовательно выводить все уникальные пары (A, B) без повторений
    - Для каждой пары запрашивать выбор пользователя: «Какой источник приоритетнее: 1 или 2?»
    - Начислять баллы: победителю +1, проигравшему +0
- После всех сравнений:
    - Отсортировать источники по сумме баллов (по убыванию)
    - Разделить отсортированный список на 5 равных частей (по количеству элементов)
    - Присвоить статусы:

| Часть | Статус | Маркер |
|:---|:---:|:---:|
| 1 (лучшие) | `highest` | 🔺 |
| 2 | `high` | ⏫ |
| 3 | `medium` | 🔼 |
| 4 | `low` | 🔽 |
| 5 (худшие) | `lowest` | ⏬ |

### 4.8. Обновление файлов
Для каждого приоритезированного источника обновить строку названия:
- Формат: `- [X] Название источника [маркер]`
    - Для `highest`: `- [/] Название источника 🔺`
    - Для `high`: `- [ ] Название источника ⏫`
    - Для `medium`: `- [ ] Название источника 🔼`
    - Для `low`: `- [ ] Название источника 🔽`
    - Для `lowest`: `- [ ] Название источника ⏬`
- Сохранить изменения в исходный файл, не затрагивая остальное содержимое

## 5. Ограничения и запреты
- Запрещено использовать внешние зависимости (только стандартная библиотека Python)
- Запрещено изменять файлы с readed: true
- Запрещено менять структуру frontmatter или другие строки файла, кроме строки названия
- Обязательно Python 3.10+ (использовать match/case при необходимости)
- Обязательно обработка ошибок: отсутствие директории, невалидный frontmatter, пустой выбор

## 6. Требования к архитектуре и коду
### 6.1. Структура кода
Рекомендуемая модульность:
```
src/
├── main.py              # точка входа, парсинг аргументов
├── parser.py            # парсинг markdown-файлов и frontmatter
├── filter.py            # фильтрация по readed/type
├── prioritizer.py       # логика двухэтапной приоритезации
├── updater.py           # обновление файлов источников
└── utils.py             # вспомогательные функции (CLI-меню, логирование)
```

### 6.2. Стиль кода
- PEP 8
- Type hints для всех функций
- Docstrings для публичных функций и классов
- Логирование через `print`

### 6.3. Архитектурные требования
- Разделение ответственности: парсинг, бизнес-логика, ввод-вывод
- Функции должны быть тестируемыми (минимум побочных эффектов)
- Интерактивные запросы к пользователю вынесены в отдельные функции

## 7. Формат результата
- Создать файлы:
    - `main.py` — CLI-интерфейс
    - `parser.py`, `filter.py`, `prioritizer.py`, `updater.py`, `utils.py` — модули логики
    - `requirements.txt` — пустой или с комментарием «no external deps»
    - `README.md` — инструкция по запуску и примеры
- Изменить: файлы источников в указанной директории (добавление маркеров приоритета)

## 8. Тестирование и критерии приёмки
### 8.1. Тесты
- Unit-тесты (pytest):
    - Парсинг frontmatter и извлечение названия
    - Фильтрация по readed и type
    - Расчёт alpha и разбиение на группы
    - Подсчёт баллов в парных сравнениях
    - Форматирование строки с маркером
- Интеграционный тест:
    - Запуск приложения на тестовой директории с 60+ файлами, проверка обновления маркеров

### 8.2. Критерии приёмки
- Запуск `python main.py --path ./sources` без ошибок на валидной директории
- Флаг `--count` корректно выводит статистику по типам
- Источники с `readed: true` не обрабатываются
- При >50 источниках работает групповой отбор, остальные получают `lowest`
- Парные сравнения выполняются без повторений, баллы считаются верно
- Итоговое распределение по 5 статусам соответствует сортировке по баллам
- В файлах источников корректно обновляется строка названия с маркером и слешем
- Приложение завершается по выбору «0» без изменения файлов

## 9. Особые указания для Code LLM
- Генерировать код пошагово, начиная с `parser.py` и `main.py`
- После каждого модуля — краткий комментарий о назначении
- В `prioritizer.py` явно прокомментировать математику разбиения на 5 частей (обработка остатка при неравном делении)
- Для интерактивного ввода использовать `input()` с валидацией выбора
- Не генерировать тесты автоматически — оставить заглушки с комментарием `# TODO: add tests`
- Все emoji-маркеры использовать в Unicode, не в виде названий
